## 1. Rubric mapping

The assignment scores on five dimensions. Here's where to look for evidence of each.

### Product Execution — 25%

- **Flow simplicity:** `/table/T1` → 2-step onboarding (name + vibe chips) → menu → cart → OTP → confirmation. New user can order in under 3 minutes; the Playwright smoke (`apps/web/tests/e2e/order-flow.spec.ts`) walks the exact path.
- **AI feels native, not bolted on:** the chat launcher lives next to the cart launcher; suggestion cards have inline "Add" buttons; the "Zara's picks for you" strip is the first thing the diner sees post-onboarding; the upsell fires from a cart event, not a button.
- **Mobile-first polish:** all tap targets ≥ 44 px (`tap-target` Tailwind class); safe-area insets respected; horizontal-scrolling category and chip rails; bottom-sheet drawers for cart and chat; PWA manifest + offline menu service worker. Lighthouse mobile target ≥ 85.

### AI Implementation — 35% (biggest bucket)

- **Eight distinct agents**, each with their own prompt, schema, persona, and golden cases — see [docs/agent-design.md](docs/agent-design.md). Source under `packages/core/src/agents/`.
- **Multi-agent coordination via a typed DAG**: `multilingualNLU → router → specialist → contextMemory`, with `sentiment` as a parallel branch and `upsell` triggered by cart events. Defined in `packages/core/src/orchestrator/graph.ts`. ADR-007 documents why we built our own DAG instead of using LangGraph.
- **RAG with hard grounding:** every Recommendation Agent response is constrained to `itemId`s drawn from a `search_menu` candidate set (pgvector cosine search). The schema validation rejects hallucinated IDs. Tested by `recommendation.golden.ts`.
- **Hinglish + Telugu-English:** first-class language detection in `multilingualNLU/`. The agent's `englishGloss` field gives downstream agents a clean intent regardless of source language; responses come back in the user's language. 8 NLU golden cases cover the matrix.
- **Tool registry with per-agent ACL:** `packages/core/src/tools/registry.ts`. `add_to_cart` and `remove_from_cart` are orchestrator-only — even a jailbroken agent can't mutate the cart directly. `sessionId` and `tableId` come from the orchestrator context, never from LLM output. This is the prompt-injection firewall.
- **Per-session LLM budget cap:** `SESSION_LLM_BUDGET_USD` in env; `sessionMemory.chargeBudget` accumulates every agent call; the orchestrator refuses to start a new turn once the cap is hit. Prevents a runaway demo bill.
- **Eval suite:** `pnpm eval` runs every agent's golden cases against the live OpenAI API, writes [`docs/eval-results.md`](docs/eval-results.md), exits non-zero on regression. CI runs this on every push to `main`.
- **Trace observability:** every agent invocation persists to `agent_traces` (input/output previews, tool calls, latency, tokens, USD cost). `/debug/trace/<sessionId>` UI renders the timeline; gated by `NEXT_PUBLIC_DEMO_MODE` (404 in prod).

### System Design — 20%

- **Clean service boundaries:** `packages/core/src/services/{menu,session,cart,otp,order,user}` — five domain services, each with `service.ts` (logic), `types.ts` (DTOs), `index.ts` (barrel). Direct Prisma reads outside these are forbidden by convention.
- **Two-process backend, one shared core:** Next.js on Vercel (API + SSE), Socket.io gateway on Render (WebSockets). Both consume the same `@smart-dining/core` workspace package — zero code duplication, zero RPC chatter. ADR-003 explains why.
- **Data model supports every feature without hacks:** snapshot fields on `order_items` so historical orders survive menu changes; `cart_items.version` for optimistic concurrency; `agent_traces` with bounded JSON previews; `users.phone_hash` as the cross-session join key with plaintext only in `orders`. Schema at `packages/core/prisma/schema.prisma`.
- **Three-tier memory (per ADR-006):**
  - Working — in-request, last 5 turns
  - Session — Redis-cached, Postgres-durable, TTL-managed
  - Long-term — phone-hash-keyed `users` table, populated at checkout, surfaces "Welcome back" on return visits.

### Performance — 10%

- **Sub-3s AI response:** intent classification on gpt-4o-mini (~300 ms), Recommendation on gpt-4o (~1500 ms), JSON-repair retry budget at 1, streaming over SSE. First token < 800 ms target met for the gpt-4o-mini path.
- **First-load JS:** 189 KB for `/table/[tableId]` (with Tailwind + Radix + TanStack Query + Socket.io client + Zustand all bundled). Under the 200 KB Lighthouse danger zone.
- **Cart sync:** Redis pub/sub → Socket.io fan-out, observed at ~150 ms cross-window in dev. The architecture's hot path is one Redis publish + one Socket.io emit; no DB read on the broadcast leg.
- **Menu read:** 60s in-memory cache in `MenuService` with explicit invalidation on (future) mutations. Cache miss is one Postgres query with an index hit.

### Code Quality — 10%

- **TypeScript strict mode** across the whole workspace, including `exactOptionalPropertyTypes` and `noUncheckedIndexedAccess`. Zero `any` types in agent code (eslint-enforced).
- **6 packages all typecheck clean** — `pnpm typecheck`.
- **29 unit tests pass** — `pnpm test`. Coverage focuses on the parts that fail silently (HMAC round-trips, preference merging, time-of-day classification, error class shapes).
- **README clarity:** [README.md](README.md). Setup is 4 commands. Manual verify steps are scripted.

## 2. What's distinctive

I think these are the things you might not see in a typical submission, ranked by how much they cost to build vs. how much they signal:

| Feature | What it shows |
| --- | --- |
| **`/debug/trace/<sessionId>` UI** | Every agent's I/O, tool calls, latency, tokens, USD cost rendered as a vertical timeline. Filter by agent. Auto-refresh. The trace IS the proof. |
| **Eval harness with per-case scoring** | `pnpm eval` against the real OpenAI API. Per-agent pass-rate, latency, cost in a Markdown table. CI catches regressions. |
| **Hand-rolled DAG with full ADR** | I started with `@langchain/langgraph` and removed it during Phase 2 once the friction-vs-benefit math turned negative. ADR-007 is the receipt — graph topology stayed, library dropped. |
| **Per-agent tool ACL** | The orchestrator-only `add_to_cart` tool means a jailbroken agent can suggest items but never directly mutate the cart. ADR's are usually about file structure; this one is about defence in depth. |
| **Long-term memory with PII boundary** | Phones are HMAC-hashed for the `users` join key; plaintext only lives in `orders.customer_phone`. LLM prompts never see plaintext PII. Return visits surface a "Welcome back, visit #N" chip. |
| **2-step micro-onboarding kills the cold-start** | Without preferences, the AI Pick strip falls back to popular items (still correct, less useful). With the vibe + allergen chips, the strip has real signal from turn 0. |
| **Mock OTP provider is blocked in production** | `OtpService.pickProvider()` refuses to construct the mock when `NODE_ENV=production`. Defence in depth, not just env validation. |

## 3. What I cut and why

I'd rather list the omissions than have you wonder.

| Cut | Reason |
| --- | --- |
| Cross-session memory **lookup on landing** | Without explicit opt-in (browser fingerprint or stored token), recognising a returning diner before they re-enter their phone has ugly privacy implications. We persist on order placement and surface on the next checkout — the spec calls long-term memory "optional", and this is the conservative read. |
| Order **status websocket** beyond `order:placed` | Kitchen `preparing → ready → delivered` transitions are wired in `OrderService.updateStatus` and broadcast to the table channel — but there's no kitchen-dashboard UI to drive those transitions. Real product would have one. |
| **Voice input** | Spec mentions it indirectly. The Web Speech API integration is ~30 lines, but I'd rather ship the depth I have than spread thin. |
| **Vector store sharding / hybrid search** | pgvector with `ivfflat(lists=8)` is correct for ~50 items. At 10k items we'd lift to ~100 lists or add Pinecone-style hybrid. ADR-002 documents the threshold. |
| **Production monitoring dashboards** | LangSmith covers agent traces; Vercel + Render dashboards cover the rest. A custom Grafana would be theatre at this scale. |

## 5. Running locally — 4 commands

```bash
pnpm install
pnpm infra:up                                # Postgres + Redis via docker-compose
cp .env.example .env                         # fill OPENAI_API_KEY at minimum
pnpm db:migrate && pnpm db:seed && pnpm dev  # web on :3000, gateway on :4000
```

Then open `http://localhost:3000/table/T1`.

## 6. Honest engineering notes

A few things a careful reviewer will see and might want context on:

- **Why hand-roll the DAG instead of using LangGraph?** Phase 2 started on it; the channel/reducer abstractions added a layer that bought us nothing at 6 nodes. ADR-007 is the postmortem.
- **Why is `add_to_cart` orchestrator-only?** Agents propose, orchestrator disposes. A jailbroken Recommendation agent can return a malicious itemId but cannot execute the mutation — the tool registry's ACL refuses. Without this, prompt injection ⇒ cart manipulation.
- **Why is the upsell rate-limited to 30s?** Without the limit, every cart add fires an LLM call. With 4 diners and a quick ordering session, that's ~16 calls in two minutes, most of which the diner doesn't even read. 30s gives one nudge per "round of decisions" — about right.
- **Why does `OrderView.visitCount` exist?** So the order confirmation can render "Welcome back — visit #N" without a second round-trip. It's the user-visible artifact of the Tier-3 memory store.
- **Why isn't there a CSRF token?** The mutating endpoints all require `sessionId` (UUID, server-issued, opaque) plus `X-Display-Name`. Without a cross-site way to learn a sessionId, classic CSRF doesn't apply. A returning-user flow would change that calculus.
